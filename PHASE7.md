# Phase 7 — real migrations, resume-driven search, company career pages

Read [[DESIGN.md]] first for the system contract; this file only holds phase 7's
step-by-step build order and rationale. See [[WORKFLOW.md]] for the recurring
process this and every phase file follows.

Same workflow rules as [[PHASE1.md]]–[[PHASE6.md]]. **Scoped but not started** —
per WORKFLOW.md rule 3, docs land first, on their own commit, before any code.
Three research threads, requested directly rather than proposed, each verified
before being written down here (WORKFLOW.md rule 2):

- **Migration tooling.** `repo._ensure_runs_model_column()` (phase 6 step 3) is
  a one-off, hand-rolled `PRAGMA table_info` + `ALTER TABLE` check — it exists
  because a real production DB (this project's own dev `scraper.db`) broke the
  moment `Base.metadata.create_all()`'s real limitation (it only creates
  missing tables, never alters existing ones) collided with a real schema
  change. That pattern doesn't scale past one column: every future schema
  change would need its own copy-pasted detect-and-patch function, with no
  single place recording *which* migrations have actually been applied to a
  given database. **Alembic** (verified real: v1.18.5 on PyPI, MIT-licensed,
  lightweight — pulls in only `mako`/`markupsafe`, nothing like `markitdown`'s
  footprint below) is SQLAlchemy's own official migration tool: versioned
  migration scripts, an `alembic_version` table tracking what's applied,
  `autogenerate` diffing real models against a real DB. Real complication,
  not hand-waved: `autogenerate` diffs against `Base.metadata`, which knows
  nothing about the `job_embeddings`/`question_embeddings` (`vec0`) or
  `job_search_fts`/`question_search_fts` (`fts5`) virtual tables from phase 6
  steps 7–8 — those were created via raw `CREATE VIRTUAL TABLE` outside the
  ORM on purpose (DESIGN.md §3, `vectors.py`/`fts.py`), and will need to stay
  raw `op.execute(...)` operations in their own migration, not autogenerated.
  Also needs real research before step 1: whether `make_engine()` should
  invoke Alembic's programmatic upgrade API at app startup (so `./run.sh`
  keeps working with zero manual steps for a single-user local tool) versus
  requiring a manual `alembic upgrade head` — the latter breaks this
  project's "one command starts everything" README promise and should be
  avoided if the programmatic API supports it cleanly.
- **Resume-driven job search.** Upload a resume PDF, convert it to Markdown,
  and use an LLM to derive the job titles/keywords worth searching for —
  verified two real PDF→Markdown libraries before picking one: **`markitdown`**
  (Microsoft, real, v0.1.6) pulls in a genuinely heavy tree for this use case
  (`onnxruntime`, `magika`, `protobuf`, `networkx` — 15 packages total, most
  of it for document types this app will never see); **`pymupdf4llm`** (real,
  v1.28.0, built on PyMuPDF) is far lighter and was tested for real against
  the user's own `resume-backend.pdf` (`~/Downloads`) — converted cleanly to
  well-structured Markdown (headings, sections, contact line) in one call,
  no tuning needed. `pymupdf4llm` is the real candidate; confirm no gaps
  against a couple more resume formats/layouts before committing to it in
  code. The "derive positions" step is a new LLM extraction task, same shape
  as `JobExtract`/`QuestionExtract` (a new Pydantic schema, validated same
  way) but producing search terms instead of stored items — needs its own
  relevance discipline (PHASE2.md step 2's precedent: a schema-validated
  LLM output that's syntactically fine can still be *wrong*, e.g. surfacing
  a role nothing in the resume supports), so the required smoke test must
  check against the user's real resume that generated positions are
  genuinely defensible from its actual content, not just present.
- **Company + career-page scraping.** A new source *direction*, not just a
  new source — every current source (jobs or questions) is a job *board*
  aggregating many companies; this scrapes one company's own career page
  directly. This resumes a thread phase 5 already researched and explicitly
  deferred: "Greenhouse/Lever/Ashby have real public per-company APIs but
  need a company-slug list, a different architecture than any current
  source" (PHASE5.md). Re-verified today, both still real and live:
  `https://boards-api.greenhouse.io/v1/boards/{slug}/jobs` and
  `https://api.lever.co/v0/postings/{slug}` both returned `200` against real
  company slugs. This may not need a new `Source` protocol shape at all —
  structurally it's the same "one JSON API, `seed_urls`/`split_items`" shape
  Arbeitnow/RemoteOK/Himalayas/RemoteJobs.org already use, just parameterized
  by a company slug instead of one fixed global endpoint. The genuinely open
  question is where the slug list comes from — a small hand-curated seed
  list (simplest, matches this project's boring-by-default bias) versus
  something dynamic; needs a real decision before step 1, not a guess. Worth
  noting: this pairs naturally with the resume-driven search thread above —
  once real target roles/companies are known, this direction becomes the
  concrete way to check whether those specific companies are hiring for them.

## Build order (draft — not started, confirm before running `/loop`)

Numbered for reference; each item above needs its own confirmation loop
(WORKFLOW.md rule 1: discuss scope before building) before this is locked in
as an actual `/loop`-driven sequence, the same way PHASE6.md's steps were
negotiated across several turns before any code landed.

1. **Alembic wired in, current schema captured as-is (backend).** Add
   `alembic` to `pyproject.toml`. Initialize, point `env.py` at `Base.metadata`
   and `config.DATABASE_URL`. Write the first real migration representing
   *today's* schema exactly (including the `runs.model` column and the
   vec0/FTS5 virtual tables as explicit `op.execute` operations) so every
   existing real database — including this project's own `scraper.db` — can
   be stamped as already at that revision without re-running anything
   destructive. Decide and implement how `make_engine()` invokes it. Smoke:
   run the new migration path against a real copy of the current dev
   `scraper.db`, confirm it lands at the expected revision with zero data
   loss and vec0/FTS5 still functional (same checks phase 6 step 7's smoke
   test used).
2. **Retire the hand-rolled migration helper (backend).** Delete
   `_ensure_runs_model_column`; any future schema change becomes a real
   Alembic migration instead of a bespoke detect-and-patch function. Smoke:
   confirm a fresh empty DB and the migrated real DB both end up with an
   identical schema.
3. **Resume upload endpoint (backend).** New `POST /api/resume` accepting a
   PDF, converts via `pymupdf4llm`, returns the Markdown (or 4xx on a file
   that isn't a real PDF / fails to parse — explicit error handling, not a
   silent empty result). Smoke: upload the user's own real resume PDF
   through the live API, confirm the returned Markdown is genuinely
   readable, not just non-empty.
4. **Resume → search-position extraction (backend).** New Pydantic schema
   (e.g. `ResumePositions`) and LLM prompt deriving job titles/keywords from
   the resume Markdown, validated the same way `JobExtract`/`QuestionExtract`
   are. Smoke: run against the user's real resume, manually confirm every
   generated position is defensible from its actual content.
5. **Resume upload UI (frontend).** A real place to upload a PDF and see the
   derived positions — likely feeding directly into the existing "New
   scrape" source picker or the search command palette, decide once step 4
   is real and there's something to wire up to. `npm run build` gate; real
   look in a browser with a real uploaded resume.
6. **One company-career-page source, proving the shape (backend).** Pick one
   real company confirmed to run Greenhouse or Lever, add it as a source
   using the existing `Source` protocol (parameterized by slug, not a new
   protocol shape, per the research above — revisit only if a real company
   doesn't fit). Smoke: one real scrape against it.
7. **Company slug list + generalizing beyond one company (backend +
   frontend).** Decide, based on step 6's real experience, how the slug list
   is sourced and surfaced in the UI. Scope this properly once step 6's
   answer is real, not guessed now.

Next: not started — propose refinements or confirm this build order before
driving it with `/loop`, per [[WORKFLOW.md]] rule 1.
